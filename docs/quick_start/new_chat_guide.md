## 📚 よく使う情報の場所

### 🔴 最重要：必読ドキュメント

**コード実装前に必ず確認してください！**

| ドキュメント | 内容 | 重要度 |
|-------------|------|--------|
| **`docs/design/design.md`** | **全体設計・コーディング規約・システムアーキテクチャ・データ構造** | ⭐️⭐️⭐️ |

**このファイルを読まないと起こる問題**:
- ❌ 予約語の使用（`owner`, `is_processing`など）
- ❌ データ構造の誤り（`ability_parsed`形式など）
- ❌ システム間の連携ミス
- ❌ 命名規則違反

**特に重要なセクション**:
- システムアーキテクチャ（全体構成図）
- 技術的制約（予約語回避、TextureRect制約）
- データ構造（JSON構造、ability_parsed）
- 開発上の重要な注意点

## ✅ チャット終了時のチェックリスト

次回のスムーズな継続のため、終了前に確認：

- [ ] **`docs/progress/daily_log.md`を更新** ← 最重要！
  - 完了した作業を記録（✅）
  - 次のステップを明記（📋）
  - 課題・メモを追記
- [ ] 課題があれば`docs/issues/issues.md`に追加
- [ ] 重要な設計変更は該当ドキュメントを更新

### daily_log.md更新テンプレート

```markdown
## YYYY年MM月DD日

### 完了した作業
- ✅ **[機能名]実装完了**
  - [詳細1]
  - [詳細2]
  - 詳細: `[ドキュメントへのリンク]`

### 次のステップ
- 📋 **[次にやること]**

### 課題・メモ
- [気づいた点など]
```

**重要**: daily_log.mdを更新しないと、次回チャットで前回の作業内容が分からなくなります！

## 🎓 チャット継続のコツ

### 1. 具体的に伝える
❌ 「前回の続きをお願いします」
✅ 「前回のチャットを引き継いでください。防御型クリーチャー21体の実装をしていました」

### 2. 作業の粒度を適切に
- 1チャット = 1〜3個の機能実装
- 大きな機能は複数チャットに分割

### 3. 重要な決定は必ずドキュメント化
- 設計変更 → `docs/design/`
- 進捗 → `docs/progress/daily_log.md`
- バグ → `docs/issues/issues.md`

### 4. 定期的に進捗確認
```
現在の進捗状況をまとめてください。
docs/README.mdを更新してください。
```

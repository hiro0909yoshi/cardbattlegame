## 概要

### 目的
GUT（Godot Unit Test）フレームワークを使い、ゲームロジックの正しさを自動検証する。
手動テストでは毎回数時間かかる検証を、Run All一発（数十秒）で完了させる。

### テストの哲学

> **テストはバグ検出器じゃなくて仕様固定装置**

- `assert_eq`（完全一致）のみ使用。`assert_true(hp <= 0)` のような緩い判定は**禁止**
- 実際のゲーム条件（ランドボーナス含む）でテスト。人工的な条件回避は**禁止**
- 数値が1でもずれたらテスト失敗。付与スキルの過不足も失敗
- 少しでも挙動が怪しいアイテムはOKリストに入れない

### 既存ツールとの関係

| ツール | 目的 | 実行方法 |
|--------|------|----------|
| **バトルテストツール** | バトル結果を目で確認 | ゲーム内UI（手動） |
| **GUT自動テスト** | ロジックの正しさを自動判定 | GUTパネルのRun All（自動） |

バトルテストツールの `BattleTestExecutor` や Mock クラスは、GUTテストでも再利用する。

### テスト方針

1. **public経由のテスト**が基本
2. 複雑なロジックは**分離してpublic化** → 単体テスト
3. ゲームフロー系（GFM, StateMachine）のprivateは**絶対にテストしない**
4. 純ロジックのprivate直接テストは**最終手段**として許容
5. メインゲームのコードは**絶対に変更しない**。テストクラス内での対応のみ。

## 環境

- **GUT バージョン**: v9.5.0（Godot 4.5.1 対応。v9.6.0はGodot 4.6専用で使用不可）
- **Godot**: 4.5.1

### 設定ファイル

**`.gutconfig.json`**:
```json
{
  "dirs": ["res://test/unit/", "res://test/battle/item/", "res://test/battle/creature/"],
  "prefix": "test_",
  "suffix": ".gd",
  "should_exit": true,
  "log_level": 1
}
```

**GUTパネル設定**: Test Directories に `res://test/unit/`, `res://test/battle/item/`, `res://test/battle/creature/` を追加

**ディレクトリ分離**: アイテムテストとクリーチャースキルテストは別ディレクトリに分離されており、GUTパネルで個別実行可能

## Phase 2: アイテムバトルテスト（完了）

### 目的
既存の `BattleTestExecutor` をGUTテストから呼び出し、アイテム効果の正しさを自動判定する。

### テスト条件（固定）

| 項目 | 値 |
|------|-----|
| 攻撃側 | タイダルオーガ（ID:138, 水属性, AP40/HP50, スキルなし） |
| 防御側 | レッドオーガ（ID:48, 火属性, AP40/HP50, スキルなし） |
| バトル土地 | タイル4（火属性, Lv1, 防御側所有） |
| ランドボーナス | 防御側のみ HP+10（火タイル×火クリーチャー＝属性一致） |
| 攻撃側ランドボーナス | なし（水クリーチャー×火タイル＝属性不一致） |

**基準値（アイテムなし）**: 攻AP=40, 攻HP=10, 防AP=40, 防HP=20, attacker_survived

### ボード配置（board_layout）

ダイアモンドボード20タイルのうち8タイルにクリーチャーを配置。

```
タイル0(CP) → 1(火)🔵 → 2(火)🔵 → 3(火)🔴 → 4(火)🔴★バトル → 5(中立)
  ↑            攻撃側     攻撃側     防御側     防御側             ↓
19(地)                                                        6(水)🔵 攻撃側
18(地)                                                        7(水)🔵 攻撃側
17(地)                                                        8(水)🔴 防御側
16(地)                                                        9(水)🔴 防御側
15(中立) ← 14(風) ← 13(風) ← 12(風) ← 11(風) ← 10(CP)
```

| プレイヤー | 所有タイル | 内訳 |
|-----------|-----------|------|
| 🔵 攻撃側(pid=0) | 4枚 | 火×2（タイル1,2）+ 水×2（タイル6,7） |
| 🔴 防御側(pid=1) | 4枚 | 火×2（タイル3,4）+ 水×2（タイル8,9） |
| 空き | 12枚 | CP×2, 中立×2, 風×4, 地×4 |

**特徴**: バトルタイル4の隣（タイル3）が防御側の土地 → `adjacent_ally_land` 条件が成立

### テスト構造

```gdscript
# 各アイテムごとに攻撃側・防御側の2テスト
func test_1009_zweihander_attacker():
	await _assert_attacker_item(1009, "ツヴァイハンダー", 90, 50, 40, -30, "attacker", [])

func test_1009_zweihander_defender():
	await _assert_defender_item(1009, "ツヴァイハンダー", 40, -40, 90, 20, "defender", [])
```

### 各テストのアサート項目（6項目）

| # | 項目 | 説明 |
|---|------|------|
| 1 | attacker_final_ap | 攻撃側の最終AP |
| 2 | attacker_final_hp | 攻撃側の最終HP（current_hp） |
| 3 | defender_final_ap | 防御側の最終AP |
| 4 | defender_final_hp | 防御側の最終HP（current_hp） |
| 5 | winner | 勝者（"attacker", "defender", "attacker_survived", "both_defeated"） |
| 6 | granted_skills | アイテム使用者に付与されたスキル（Array） |

### HP消費順序（ダメージ処理）

```
land_bonus → resonance → temporary → spell → item → current_hp
```

`final_hp` は `current_hp` の値。アイテムHP吸収分は含まれない。

### 術攻撃（巻物）の特性

- 術攻撃はランドボーナスを**貫通**する（防御側base HPに直接ダメージ）
- 巻物APが使用者の攻撃APとして記録される

## Phase 3: クリーチャースキルテスト（実装中）

### 目的
各クリーチャー固有スキルの発動と効果を自動検証する。

### テスト条件

アイテムテスト（Phase 2）と異なり、**クリーチャーごとに対戦相手とバトル土地が変わる**。
各テストで `_create_config(attacker_id, defender_id, defender_land)` により個別設定する。

### テスト構造

```gdscript
# test/battle/creature/test_battle_creatures.gd
extends GutTest

var _executor: BattleTestExecutor

func before_all():
	_executor = BattleTestExecutor.new()
	_executor.scene_tree_parent = self

func _create_config(attacker_id: int, defender_id: int, defender_land: String = "neutral") -> BattleTestConfig:
	var config = BattleTestConfig.new()
	config.attacker_creatures = [attacker_id]
	config.defender_creatures = [defender_id]
	# defender_landに応じたタイルインデックスを設定
	var tile_idx: int
	match defender_land:
		"fire": tile_idx = 4
		"water": tile_idx = 6
		"wind": tile_idx = 11
		"earth": tile_idx = 16
		_: tile_idx = 5  # neutral
	config.battle_tile_index = tile_idx
	config.board_layout = [
		{"tile_index": tile_idx, "owner_id": 1, "creature_id": defender_id},
	]
	config.attacker_battle_land = defender_land
	config.defender_battle_land = defender_land
	config.attacker_battle_land_level = 1
	config.defender_battle_land_level = 1
	return config
```

### ステータス上昇系スキルの共通テストルール

AP上昇を伴うスキル（共鳴、土地数比例、強打など）は、基本テストに加えて以下2パターンを必ずテストする。

#### 1. 強化アイテム併用テスト

スキルによるAP上昇後に強化（×1.5）が正しく掛かるかを確認する。

```
処理順: アイテムAP加算 → スキルAP上昇 → 強化×1.5
例: AP30 +30(item) +20(共鳴) = 80 → ×1.5 = 120
```

- 強化がスキル上昇前ではなく**上昇後**の値に掛かることを検証
- 使用アイテム例: ジャイアントキラー(1060, AP+30, 強化[MHP>=40])

#### 2. 巻物アイテム併用テスト

スキルによるAP上昇が巻物APに影響しないことを確認する。

```
処理順: 巻物AP固定(40) → スキルAP上昇(+20→60) → 術攻撃チェック(バフ検出→AP40にリセット)
結果: 巻物APは固定値のまま。HPボーナスは有効。
```

- 巻物APは**固定値**であり、スキルのAP上昇の影響を受けない
- HPボーナスは巻物使用時も有効（resonance_bonus_hpとして吸収に使われる）
- 使用アイテム例: ライトニングオーブ(1024, 術攻撃[AP40])

### バグ修正記録

**`skill_first_strike.gd` grant_skill() の不具合**:
- **問題**: `grant_skill("先制")` が `has_first_strike = true` のみ設定し、`has_item_first_strike` を設定していなかった
- **影響**: アイテムによる先制付与が後手スキルを上書きできなかった
- **修正**: `participant.apply_item_first_strike()` を呼び出すように変更。`has_item_first_strike = true` と `has_last_strike = false` を正しく設定

**`battle_test_executor.gd` first_strike_occurred 未設定**:
- **問題**: `BattleTestResult.first_strike_occurred` フィールドは存在したが、Executor側で値を設定していなかった
- **修正**: `determine_attack_order()` 後に先制判定ロジックを追加
